# llming-docs — API design

This is the canonical contract for how documents are created, edited,
read, and transported in llming-docs. **One vocabulary for every doc
type, one shape for every operation, one transport function for every
WebSocket host.** When in doubt, follow these rules.

## 1. One op vocabulary across `create_document` and `update_document`

**Creation = empty doc + operations.** There is no per-type create-only
payload. The same `operations` array that updates a document is what
populates it at create time.

```json
// text_doc
{
  "tool": "create_document",
  "type": "text_doc",
  "name": "Q1 Report",
  "operations": [
    {"op": "add", "path": "sections/-",
     "value": {"id": "s1", "type": "heading", "level": 1, "content": "Q1 Report"}},
    {"op": "add", "path": "sections/-",
     "value": {"id": "s2", "type": "paragraph", "content": "..."}}
  ]
}

// table
{
  "tool": "create_document",
  "type": "table",
  "name": "Q1 Sales",
  "operations": [
    {"op": "set", "path": "sheets/0/name", "value": "Q1"},
    {"op": "bulk_set", "path": "sheets/0/range/A1", "values":
      [["Product", "Revenue"], ["Widget", 100], ["Gadget", 200]]},
    {"op": "set", "path": "sheets/0/cells/B1/font", "value": {"bold": true}}
  ]
}
```

`update_document` accepts the same `operations` against an existing doc.

**Why this matters:** the LLM learns one op vocabulary per doc type and uses
it for everything — drafting from scratch, editing later, fixing mistakes.
The wheel isn't reinvented per type.

**Implementation rule:** when adding a new doc type, define the op
dispatcher *first*. `create_document` for that type is then "create empty +
apply ops" with zero new code. If you find yourself reaching for a typed
"initial payload" parameter, stop — extend the op vocabulary instead.

## 2. Storage is the canonical format, not JSON

Each doc type chooses the storage format that matches its richness:

| Type           | Storage                       | Why                                           |
|----------------|-------------------------------|-----------------------------------------------|
| `text_doc`     | JSON `{sections}`             | Native to our editor; lossless                |
| `table`        | **XLSX bytes (openpyxl)**     | Lossless support for borders / fills / formulas / merges / charts |
| `presentation` | JSON                          | (TODO: PPTX-native, same pattern as table)    |
| `plotly`       | JSON                          | Native to Plotly                              |
| `html`         | HTML string                   | Native                                        |

**No JSON façade for binary-backed types.** A `table` doc holds an openpyxl
`Workbook` server-side; the persisted form is XLSX bytes (base64 in our doc
store / IndexedDB). There is no `{sheets: [...]}` JSON anywhere in storage —
not in the doc, not in IDB, not in conversation snapshots.

JSON appears only at three boundaries:

1. The op vocabulary (LLM / client → server).
2. The read API responses (server → LLM).
3. The render `view` attached to `data` for client display (server → frontend).

These responses are generated on demand from the canonical storage. They
are never persisted as the source of truth.

### The `view` field (client-side rendering of binary-backed types)

When a doc event is emitted to the client (`doc_created`, `doc_updated`),
binary-backed types get a derived render snapshot attached as
`document.data.view`. For `table` this is a JSON view of the workbook
(`{sheets: [{name, columns, rows, cells?, column_widths?, row_heights?,
merges?, freeze_panes?}]}`) so the browser can render styled cells without
vendoring an XLSX JS library.

The view is:
- generated by `llming_docs.client_payload.client_doc_payload(doc)`
- never persisted as canonical storage
- regenerated on every emit (cheap — sparse cells dict)

## 3. Read API — three tools, llming-flux-compatible where applicable

### `read_document(document_id)`
Universal "what's this doc?" overview. Type-discriminated response shape.

For `text_doc`:
```json
{ "type": "text_doc", "name": "...", "section_count": 5,
  "sections": [{"id": "s1", "type": "heading", "preview": "..."}] }
```

For `table`:
```json
{ "type": "table", "name": "...",
  "sheets": [{"name": "Q1", "max_row": 50, "max_col": 5,
              "headers": ["Product", "Revenue", ...],
              "sample_rows": [{"Product": "Widget", "Revenue": 100}, ...]}] }
```

Cheap orientation call; the LLM uses this first, then drills in.

### `query_table(document_id, sheet?, range?, columns?, filters?, limit?, offset?)`

XLSX-only. Returns the **same shape llming-flux's `query_table` returns**:

```json
{
  "table_name": "Q1",
  "headers": ["Product", "Revenue"],
  "rows": [{"Product": "Widget", "Revenue": 100}, ...],
  "total_rows": 50,
  "returned_rows": 20
}
```

- `sheet`: name or 0-based index. Defaults to the first sheet.
- `range`: A1-style (`"A1:E50"`) or omitted (whole sheet).
- `columns`: `["Product", "Revenue"]` — projection by header name.
- `filters`: `{"Region": "North"}` — case-insensitive substring match per
  column. Matches llming-flux semantics exactly.
- `limit` / `offset`: pagination. Default `limit: 20`, max `100`.

Values only — no formatting. For formatting, use `query_cells`.

### `query_cells(document_id, sheet, range, with_formatting=true)`

XLSX-only. Sparse cell-level access — the only API that exposes per-cell
formatting (font, fill, alignment, border, number format).

```json
{
  "sheet_name": "Q1",
  "range": "A1:B3",
  "cells": {
    "A1": {"value": "Product", "font": {"bold": true}},
    "B1": {"value": "Revenue", "font": {"bold": true},
           "alignment": {"horizontal": "right"}},
    "B2": {"value": 100, "number_format": "$#,##0"}
  }
}
```

Cells without explicit formatting are absent (sparse). Use this when
inspecting or replicating styles, not for bulk data reads.

## 4. Op vocabulary by type

### Op kinds (universal across types)

| Op         | Semantics                                                           |
|------------|---------------------------------------------------------------------|
| `set`      | Replace the value at `path`.                                        |
| `add`      | Insert into a list (with `position` or `path/-` for append) OR create a new key on a dict. |
| `remove`   | Delete the item / key at `path`.                                    |
| `move`     | Move an array item to a different `position` (rare).                |
| `bulk_set` | XLSX-only sugar — drop a 2D array starting at a top-left cell. Expands to per-cell `set`s internally. |

### `text_doc` ops
- `set sections/<sid>/<field>` — replace a field on a section
- `add sections/-` — append section
- `add sections/<index>` (with `position`) — insert at index
- `remove sections/<sid>` — delete by id

### `table` ops (openpyxl-native paths)

Cell-level (most common):
- `set sheets/<i>/cells/<addr>/value` — set a cell value (e.g. `cells/B3/value`)
- `set sheets/<i>/cells/<addr>/font` — `{bold, italic, color, size, name, underline, strike}`
- `set sheets/<i>/cells/<addr>/fill` — `{fill_type, start_color, end_color}`
- `set sheets/<i>/cells/<addr>/alignment` — `{horizontal, vertical, wrap_text, shrink_to_fit, indent, text_rotation}`
- `set sheets/<i>/cells/<addr>/border` — `{top, right, bottom, left, diagonal}` (each side: `{style, color}`)
- `set sheets/<i>/cells/<addr>/number_format` — string e.g. `"$#,##0.00"`
- `set sheets/<i>/cells/<addr>` — bundle: any subset of the above in one op

Row / column / sheet structure:
- `add sheets/<i>/rows/-` — append row (value: `[v1, v2, ...]`)
- `add sheets/<i>/rows` (with `position: <r>`) — insert row at 1-based row number
- `remove sheets/<i>/rows/<r>` — delete row r (1-based; row 1 is the header)
- `add sheets/<i>/columns/-` — append column header
- `add sheets/<i>/columns` (with `position: <c>`) — insert column at 1-based index, shifts data right
- `remove sheets/<i>/columns/<col_letter_or_index>` — delete column (e.g. `columns/B`)
- `set sheets/<i>/columns/<col>/width` — set column width
- `set sheets/<i>/rows/<r>/height` — set row height
- `add sheets/-` — append a sheet (value: name string, or null/{} for default)
- `remove sheets/<i>` — delete sheet (cannot remove last)
- `set sheets/<i>/name` — rename
- `add sheets/<i>/merges/-` — value: `"A1:B2"` style range
- `set sheets/<i>/freeze_panes` — value: `"A2"` etc.

Bulk shortcut (sugar over per-cell ops):
- `bulk_set sheets/<i>/range/<top_left>` — value: `[[...], [...]]` 2D array;
  expands to per-cell sets starting at `<top_left>`. Saves tokens for
  drafting large tables.

Address format: openpyxl-native A1 notation. `B3` means column 2, row 3.
Row 1 is conventionally the header row (first row of `query_table`'s
`headers`); data rows start at row 2.

## 5. WebSocket transport (host integration)

All client-side editing flows through `llming_docs.transport`. This is the
single integration surface for any host that wants to embed the editor —
lodge today, a Quasar app tomorrow.

### Wire protocol — incoming (client → server)

| Message                                                          | Effect |
|------------------------------------------------------------------|--------|
| `{type: "client_doc_create", doc_type, name, operations?}`       | Create a doc; same vocabulary as `create_document` MCP tool. |
| `{type: "client_doc_update", document_id, operations, name?}`    | Apply ops to an existing doc. |
| `{type: "client_doc_undo",   document_id}`                       | Pop the doc store's history one step (server tracks). |
| `{type: "client_doc_redo",   document_id}`                       | Step forward through the redo stack. |
| `{type: "client_doc_delete", document_id}`                       | Remove the doc. |

All messages may carry an optional `request_id` echoed back in `client_doc_error`
responses.

### Wire protocol — outgoing (server → client)

| Message                                              | When                                         |
|------------------------------------------------------|----------------------------------------------|
| `{type: "doc_created", document: {…}}`               | Doc created (any source: LLM, client, code). |
| `{type: "doc_updated", document: {…}}`               | Doc data / name changed.                     |
| `{type: "doc_deleted", document: {…}}`               | Doc removed.                                 |
| `{type: "client_doc_error", request_id?, errors:[…]}`| An incoming `client_doc_*` failed.           |

The `document` payload is `client_doc_payload(doc)` — `doc.model_dump()`
plus, for binary-backed types like `table`, a derived `data.view` for
rendering.

### Server-side integration (host code, e.g. lodge)

```python
from llming_docs.transport import handle_client_doc_message, make_doc_notify

# Per session, at WS connect time:
store = entry.doc_manager.store              # one DocumentSessionStore per session
store.set_notify_callback(make_doc_notify(controller._send))  # bind WS

# Per inbound message:
async def on_ws_message(msg: dict):
    if await handle_client_doc_message(store, msg, controller._send):
        return       # claimed by llming-docs
    # …route other host-specific messages…
```

That's the entire surface. Lodge adds approximately twelve lines for the
editor; a Quasar host adds the same twelve lines.

### Isolation contract — load-bearing

Doc data MUST NOT leak across user sessions. The transport guarantees this
**by being stateless about sessions**:

- No globals keyed by session, document_id, or user.
- No "look up store by id" function.
- No shared cache.
- Every operation runs against the `store` argument; every event goes
  through the `send` argument.

The host is responsible for:

1. Maintaining one `DocumentSessionStore` per user session.
2. Passing **that session's** store + **that WS's** send for every message.
3. Never sharing stores across sessions.

If the host does that correctly, cross-session leakage is impossible by
construction. There is no code path through transport that could reach
another session's data because transport doesn't know other sessions exist.

Tests pinning this contract live in `tests/test_transport_isolation.py`.
A test specifically guards against future refactors that might add a
session registry — it scans `transport.py` for forbidden global names
(`_stores`, `_sessions`, `_session_stores`, …) and fails the build if
they appear.

## 6. Persistence and migration

**Legacy table docs stored as `{sheets: [...]}` JSON** are loaded
transparently: on first access, the JSON is converted to an openpyxl
workbook and the doc's `data` is rewritten as `{xlsx_b64: "..."}`. The
JSON shape is gone after that one save.

Two migration boundaries trigger the conversion:

* `DocumentSessionStore.restore_from_list(...)` — host restores client-
  cached docs from IDB on conversation reload.
* `DocumentSessionStore.create(...)` — guards against direct legacy
  payloads from older client code (e.g. the workspace `+ new doc`
  template path).

There is also a defensive in-line migration in
`ops_dispatcher._apply_table_ops`: if a `client_doc_update` reaches the
server with a still-legacy doc (race against `restore_from_list`), the
op is applied to a freshly-migrated workbook and the doc emerges in the
new shape with the user's edit included.

The migration helper itself lives in `llming_docs.sheet.xlsx_migrate`. It
is **not** a runtime bridge — only a one-shot conversion.

## 7. Undo / redo

Server-side history is the source of truth for both undo and redo. The
client only tracks two counters per doc — "how many of MY edits can I
still undo / redo" — and fires `client_doc_undo` / `client_doc_redo` on
Cmd+Z / Cmd+Shift+Z.

`DocumentSessionStore` provides:
- `undo(doc_id)` → pops history one step, pushes the popped current state
  to a redo stack so it can be restored later.
- `redo(doc_id)` → pops the redo stack, re-records the current state on
  the undo stack (so a subsequent undo cleanly reverts the redo).

Every fresh edit (from the LLM, the client, or any code path that calls
`store.update`) clears the redo stack — same semantics as every text
editor's "any new edit invalidates the forward branch".

For binary-backed types like `table`, this means undo/redo revert
`xlsx_b64` bytes wholesale. That's bulletproof — no inverse-op
reconstruction, no possibility of "undo this column delete" failing
because we couldn't faithfully recreate the column. Round-trip tested
in `tests/test_xlsx_integration.py::TestUndoRedo`.

### AI boundary

Per UX rule: undo and redo must NOT cross AI-driven edits. The frontend
enforces this by resetting both counters to 0 whenever an AI streaming
update lands on the doc (`doc_streaming_end` for that doc id). The
server-side history still holds AI changes (the LLM uses them via
`undo_document`), but the client's UI counters cap the user's reach.

## 8. Things that are NOT supported (yet)

- Cross-sheet joins
- Server-side formula evaluation (formulas round-trip as text but render
  as the formula expression in the UI; user opens the XLSX in Excel /
  LibreOffice to compute)
- SQL syntax — we deliberately mirror llming-flux's tool-based query API
  instead. Adding raw SQL later is fine; doing both at once is not.

## 9. When in doubt

- Need a new way to mutate a doc? **Add an op**, not a new tool.
- Need a new way to read a doc? **Extend `query_table` / `query_cells`** with
  more parameters; only add a new tool if the response shape is genuinely
  different.
- Want a per-type "create with payload" shortcut? **Don't.** Document the
  ops the LLM should use to populate the doc.
- Want to add a session registry to `transport.py`? **Don't.** Pass the
  store in. The isolation contract depends on this.
- Want client-side undo to reconstruct deleted state via inverse ops?
  **Don't.** Use the server's history; it's already there.
